# git-commit

This project keeps a record of **how** each change was produced — the prompt that drove it and the
model that wrote it — so future readers (human or agent) can reconstruct intent from `git log` alone.
A bare `git commit -m "fix stuff"` throws that context away. This skill standardizes a verify-then-commit
cycle that bakes provenance into every message.

## When to use

Any time work is ready to be recorded: the user says "commit", "commit this", "save to git", or you've
just completed a prompt → verify → commit loop. Use it even for small commits so the history stays uniform.

## Workflow

1. **Verify before committing.** Don't commit blind. Confirm the change actually does what was asked:
   - Run the project's tests / build / typecheck if they exist and are relevant to the change.
   - For docs or config-only changes, a quick read-back of the diff is enough.
   - If verification fails, fix it (or tell the user) *before* committing — never commit known-broken work.

2. **Review what you're staging.** Run `git status` and `git diff` (or `git diff --staged`). Stage only
   the files relevant to this unit of work — don't sweep in unrelated untracked files (build output,
   editor dirs like `.idea/`, local scratch). Prefer explicit paths over `git add -A`.

3. **Identify the provenance inputs:**
   - **Original prompt** — the user's verbatim request that drove this change. If several messages shaped
     it, use the one that defined the task; quote it exactly, don't paraphrase.
   - **Changes** — a short bullet list (2–6 bullets) of what actually changed, at the file/feature level.
   - **Model** — the model powering *this* session. Read it from your environment/system context and
     write it as `Family X.Y (model-id)`, e.g. `Opus 4.8 (claude-opus-4-8)`. Do not hardcode a model —
     report whichever model is actually running.

4. **Commit with the template below**, always via `git commit -F -` with a heredoc so the multi-line
   body and the verbatim prompt survive intact (avoid stacking many `-m` flags).

5. **Confirm** with `git log --oneline -1` and report the resulting hash to the user.

### Branch safety

Follow the repo's policy: commit only when the user asks. If the user wants a PR or you're about to
commit to the default branch and they haven't said to commit directly here, branch first. If the user
has been committing straight to the current branch in this session, continue matching that.

## Commit message template

```
<type>(<scope>): <imperative subject, ~50 chars>

<1–3 sentence summary of the change and why.>

Changes:
- <bullet: what changed>
- <bullet: what changed>
- <bullet: what changed>

Generated by Claude Code.
Model: <Family X.Y> (<model-id>)

Prompt used:
  <the user's original prompt, verbatim, indented two spaces>

Co-Authored-By: Claude <Family X.Y> <noreply@anthropic.com>
```

- `<type>` follows Conventional Commits: `feat`, `fix`, `docs`, `chore`, `refactor`, `test`, etc.
- Indent the prompt body two spaces so it reads as a block and won't be mistaken for message structure.
- If the prompt is long, keep it complete anyway — provenance is the whole point; don't truncate it.
- If multiple prompts genuinely drove one commit, use a `Prompts used:` heading with `Prompt 1:` /
  `Prompt 2:` blocks (as in this repo's root-commit README history).

## Example

**Example 1**

Input prompt: `add a healthcheck endpoint that returns 200 and the git sha`
Verification: ran `pnpm test` — passing.

Commit:

```
git commit -F - <<'EOF'
feat(api): add /healthz endpoint returning sha

Adds a lightweight health check for uptime probes and deploy verification.
Returns 200 with the current git sha so we can confirm which build is live.

Changes:
- src/routes/healthz.ts: new GET handler returning { status, sha }
- src/app.ts: register the healthz route
- tests/healthz.test.ts: assert 200 + sha shape

Generated by Claude Code.
Model: Opus 4.8 (claude-opus-4-8)

Prompt used:
  add a healthcheck endpoint that returns 200 and the git sha

Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>
EOF
```

## Why this shape

- **Prompt first-class, not paraphrased** — the exact words capture intent and ambiguity that a tidy
  summary would smooth over; that's often what you need months later.
- **Bullet changes** — let a reader grasp scope without reading the diff.
- **Model recorded** — different models produce different work; knowing which one helps when revisiting
  or debugging a change, and lets you spot model-specific patterns across history.
